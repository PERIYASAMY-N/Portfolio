# Complete Portfolio CMS Audit & Debug Report

## 1. DOM Audit Findings
Our thorough inspection of the Admin Dashboard highlighted the exact structure and IDs being used for the **Project Form**:
*   **Form Structure**: Created dynamically via JavaScript (`vProjects.openModal`) — missing a traditional `<form>` wrapper to rely on raw input selections.
*   **Title Input**: `id="pr-title"`
*   **Description Area**: `id="pr-desc"`
*   **GitHub/Live URL**: `id="pr-git"` / `id="pr-live"`
*   **Date Input**: `id="pr-date"`
*   **Featured Input**: `id="pr-feat"`
*   **Technologies Input**: `id="pr-tech"`
*   **File Selection Input**: `id="pr-img-file"`
*   **Image State Input**: *CRITICAL FAILURE POINT* (Expected `id="pr-img"`, but it was completely missing from the modal body).

## 2. Event Listener & Form Issues Found
### Issue #1: Project Save Button Silent Failure (JS Error)
**Symptom:** Hovering and clicking the "Save" button in the "Add Project" / "Edit Project" form did absolutely nothing, and no network request was triggered.
**Root Cause:**
Inside the `vProjects.save(isEdit)` event listener logic, the function was halting abruptly due to a **TypeError**:
```javascript
// This instruction threw an error: Cannot read properties of null (reading 'value')
const imgUrl = document.getElementById('pr-img').value; 
```
The element `pr-img` was not injected into the Project DOM block. Because this threw an exception synchronously before the API call (`api.admin.updateProject`) was triggered, JS execution aborted, and the modal failed to close.
**Fix Implemented:** Target instruction removed entirely. The project form payload constructs correctly via `vProjects.currentImages` without relying on that buggy DOM query. The click event executes down through `POST / PUT` with valid data mapping to DTOs.

## 3. Image URL & Cache Busting Issues Found
### Issue #2: Public Portfolio Not Registering Uploaded Modifications
**Symptom:** An admin reliably uploads a new Profile/Project/Hero image. The file transmits successfully across `FormData`, the backend processes the `MultipartFile` properly saving to the filesystem/database, but the frontend maintains the outdated image.
**Root Cause:**
Browser caching. The frontend requests image records directly using statically pathed strings generated by the backend (e.g., `/uploads/profile/image.png`). When an entity replaces the file under the prior namespace architecture, the frontend skips the network request and loads from memory bounds.

**Fix Implemented:** 
*   **Admin Dashboard (`dashboard.js`)**: Appended cache-busting timestamp mechanisms `?t=[Date.now()]` onto `img.imageUrl` in the admin image previews (`renderPreviews()`), ensuring the administrator interface updates immediately tracking newly uploaded binaries.
*   **Public Frontend (`main.js`)**: Addressed the systemic update propagation flaw uniquely by universally amending `getMediaUrl(url)`:
    ```javascript
    function getMediaUrl(url) {
      if (!url) return '';
      const base = url.startsWith('http') ? url : 'http://localhost:8080' + url;
      return base + '?t=' + Date.now();
    }
    ```
    Every image generated natively bypasses memory caches forcing direct fetches, successfully fixing rendering for Dashboard Images matching Profile, Project, Certification, and Achievements universally!

## 4. Public DOM Rendering & Placeholder Issues Found
### Issue #3: Fallback Placeholders Covering Images or Rendering Blanks
**Symptom:** During asynchronous image load errors (or database misalignments causing a 404 image link output), placeholders were either statically bound under `style="display: none;"` incorrectly or failing to overlay as fallbacks natively.
**Root Cause:**
No intrinsic `onerror` error recovery boundaries were appended onto elements directly consuming `<img src="..." />` tags or `style.backgroundImage` references.

**Fix Implemented:**
*   Rebound the logic injecting the frontend `main.js` `DOM elements` utilizing `.onerror` protocols seamlessly resolving missing or broken 404 images to show up as placeholders reliably guaranteeing visibility logic mapping matches conditions (`naturalWidth > 0` validation analogue completion).
*   For `backgroundImage` elements dynamically handling hero images, transitioned onto constructing an in-memory `new Image()` buffer to pre-validate image presence `img.onload()` vs `img.onerror()` prior to merging stylistic rules.
